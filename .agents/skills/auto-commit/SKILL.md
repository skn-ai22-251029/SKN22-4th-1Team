## 🎯 트리거 조건
안티그래비티 (또는 다른 AI 에이전트)는 다음 상황에서 이 Skill을 **자동으로 읽고 실행**해야 합니다:
1. 사용자가 요청한 코드 수정, 버그 수정, 새 기능 구현 작업 등을 에이전트가 막 끝냈을 때
2. 사용자가 직전에 "다 했어?" 하거나, 에이전트가 "작업을 완료했습니다"라고 답해야 하는 시점

## 🛠️ 실행 규칙 (User Confirmation Flow)

에이전트가 코드를 다 고쳤다면, **절대 바로 커밋을 실행하지 말고**, 다음과 같은 뉘앙스의 메시지로 사용자에게 물어봐야 합니다:

### 1️⃣ 사용자에게 묻는 유도성 질문 포맷
> "✅ 요청하신 코드 수정을 완료했습니다! 정상 작동하는지 한번 확인해 주시겠어요?
> 
> 정상적으로 작동한다면 **'커밋 해줘'** (또는 '좋아', '완료')라고 말씀해 주세요. 제가 적절한 커밋 메시지를 작성해서 자동으로 커밋하겠습니다.
> 만약 커밋을 원치 않으시면 **'커밋 하지 마'** 라고 말씀해 주시면 진행하지 않겠습니다."

### 2️⃣ 사용자의 응답에 따른 처리
- **사용자가 "커밋 해줘", "ㅇㅇ", "좋아" 등 긍정의 대답을 한 경우**:
  - `git add .` (수정된 파일들을 스테이징)
  - `git commit -m "[Conventional Commits 규칙에 맞춘 메시지 자동 생성]"`
  - 커밋이 성공했음을 알리고, **바로 이어서 [auto-pr] 스킬과 연동하여 PR 생성을 제안합니다.** (예: "커밋이 완료되었습니다! 이 내용을 바탕으로 바로 대상 브랜치에 PR을 올릴까요?")
- **사용자가 "커밋 하지 마", "아니", "추가로 더 고칠게" 등의 대답을 한 경우**:
  - **절대 커밋을 진행하지 않습니다.**
  - 사용자의 다음 지시(추가 수정 등)를 대기합니다.

### 📝 커밋 메시지 작성 규칙 (Conventional Commits & Issue 연동)
커밋을 진행할 때는 다음 단계를 거쳐 메시지를 자동 생성하세요.

**1. 관련 이슈 확인**: `gh issue list --assignee "@me"` 또는 현재 맥락을 통해 현재 작업과 관련된 열린 이슈 번호(예: `#3`)가 있는지 파악합니다.
**2. 메시지 포맷팅**: 다음 접두어(Prefix) 중 하나를 사용해 제목을 작성하고, 본문에 이슈 번호를 포함합니다.
**⚠️ 매우 중요: 접두어 이후의 설명 메시지(제목 및 본문)는 반드시 '한국어'로 작성해야 합니다.**

*(예시)*
```text
feat: 로그인 버튼 파란색으로 변경

- 관련 이슈: #3
```

- `feat:` : 새로운 기능 추가
- `fix:` : 버그 수정
- `docs:` : 문서 수정
- `style:` : 코드 포맷팅, 세미콜론 누락 등 코드 변경이 없는 경우
- `refactor:` : 코드 리팩토링
- `test:` : 테스트 코드, 리팩토링 테스트 등 추가
- `chore:` : 빌드 업무 수정, 패키지 매니저 수정 등
