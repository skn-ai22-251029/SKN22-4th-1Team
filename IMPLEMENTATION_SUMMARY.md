# 구현 변경 요약

작성일: 2026-03-01

## 1) 응답 플로우/구조 조정
- 증상 응답 플로우를 `초기 응답(한국 DB/DUR)` + `후속 비동기 보강(미국 OTC/FDA 요약)` 구조로 재정리.
- 증상 결과 페이지에서 프론트 비동기 호출(`/api/symptom-products/`)로 제품/미국 DUR 정보를 후반 병합.

## 2) 성분 추출/선정 로직 개선
- 증상 검색은 한국 DB(`unified_drug_info.efficacy`) 기반으로 수행.
- DB 매칭 결과에서 성분 후보를 점수화하고 상위 성분을 선택.
- `ingr_kor_name` 기반이 아닌 `main_ingr_eng`/`ingr_eng_name` 중심으로 조회 흐름 정리.
- 최종 렌더 순서를 **DB 순위(1~10)** 기준으로 고정:
  - 초기 노출: 1~5
  - OTC 제품 미존재 시: 6,7,8... 순위 성분으로 대체

## 3) can_take 정책 정리
- 사용자 프로필(복용약/알레르기/기저질환) 정보가 없는 경우 기본적으로 `can_take=true` 처리.
- `can_take=false`는 사용자 개인 위험정보와 DUR 충돌이 있을 때만 사용하도록 정책 반영.

## 4) 캐시 로직 제거
- `search_cache` 조회/저장 분기 제거.
- 증상 응답은 항상 실시간 계산으로 동작하도록 변경.
- 캐시 테이블 미존재로 발생하던 관련 에러 로그 경로 제거.

## 5) FDA 조회/표시 동작 개선
- 성분별 OTC 제품 조회와 FDA 경고 요약을 비동기 API에서 처리.
- 제품 없는 성분은 결과에서 숨김 처리.
- 제품이 확인된 성분 최대 5개만 노출.

## 6) 템플릿(UI) 정리
- DUR 강제 확인 팝업(오버레이) 제거.
- 결과 페이지에서 다음 요소 제거:
  - `성분별 주의사항` 블록
  - `중요 항목 요약` 블록
- 우측 패널은 `성분별 안내사항`만 표시하도록 정리.

## 7) 현재 핵심 동작 규칙
1. 사용자 증상 입력
2. 한국 DB에서 증상 매칭 후 성분 후보 추출/랭킹
3. DB 순위 기준 상위 성분(1~10) 구성
4. 초기 화면은 1~5 성분 우선 표시
5. 비동기로 OTC/FDA 조회 후, OTC 없는 성분은 6~10 순위로 대체
6. 최종적으로 제품 확인된 성분 최대 5개 노출
